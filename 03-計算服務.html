計算服務

計算服務是 AWS 的核心服務之一，SAP 考試中涉及 EC2、Auto Scaling、ELB、容器服務等關鍵知識點，以及如何設計高可用、可擴展的計算架構。

EC2 (彈性計算雲)

實例類型和系列

  • 通用型 (T, M)
    • 平衡計算、內存和網絡資源
    • T 系列：可突增性能，適合流量波動的工作負載
    • M 系列：平衡的計算和內存，適合中型數據庫和內存處理
  • 計算優化型 (C)
    • 高性能處理器，高計算密度
    • 適合批處理、科學建模、高性能 Web 服務器
  • 內存優化型 (R, X, z)
    • R 系列：適合內存密集型應用
    • X 系列：極高內存，適合大型數據庫
    • z 系列：高頻內存，適合電子設計自動化
  • 存儲優化型 (D, H, I)
    • 高磁盤吞吐量，大量本地存儲
    • 適合數據倉庫、分佈式文件系統
  • 加速計算型 (P, G, F)
    • GPU/FPGA 加速
    • 適合機器學習、圖形處理、視頻編碼

考試要點：了解不同實例類型的特點和適用場景，能夠為特定工作負載選擇最合適的實例類型。

購買選項

  • 按需實例
    • 按小時付費，無長期承諾
    • 最靈活，但成本最高
    • 適合短期、不可預測或波動的工作負載
  • 預留實例
    • 預付費用換取折扣，1-3 年期限
    • 標準 RI：最高折扣，但靈活性最低
    • 可轉換 RI：允許更改實例類型，但折扣較低
    • 計劃 RI：在指定時間段內預留容量
    • 適合穩定、可預測的工作負載
  • Savings Plans
    • 承諾使用量換取折扣
    • 計算 Savings Plans：適用於 EC2、Fargate 和 Lambda
    • EC2 Instance Savings Plans：僅適用於 EC2
    • 比 RI 更靈活
  • 競價型實例
    • 利用備用容量，價格大幅降低但可能被中斷
    • 適合容錯、無狀態、靈活的工作負載
    • 可以設置最高價格
    • 可以使用 Spot Fleet 管理多個實例類型
  • 專用實例
    • 在專用硬件上運行，與其他客戶隔離
    • 可能需要滿足合規或許可要求
    • 可以與 RI 結合使用
  • 專用主機
    • 完全專用的物理服務器
    • 可見底層物理核心
    • 適合帶有核心許可的軟件
    • 支持主機親和性

成本優化：為不同類型的工作負載選擇合適的購買選項可以顯著降低成本。例如，對於穩定的基礎負載使用 RI 或 Savings Plans，對於可中斷的工作負載使用競價型實例，對於峰值負載使用按需實例。

實例存儲選項

  • 實例存儲
    • 臨時塊級存儲，實例終止時數據丟失
    • 物理連接到主機
    • 高 I/O 性能
    • 適合臨時數據、緩存、臨時文件
  • EBS (彈性塊存儲)
    • 持久塊級存儲，獨立於實例生命週期
    • 可以在實例之間移動
    • 支持快照和加密
    • 多種卷類型適合不同工作負載

實例元數據和用戶數據

  • 實例元數據
    • 關於實例的信息，可通過 http://169.254.169.254/latest/meta-data/ 訪問
    • 包括實例 ID、安全組、MAC 地址等
    • 可用於自動化腳本和應用程序
  • 用戶數據
    • 啟動時執行的腳本，可通過 http://169.254.169.254/latest/user-data/ 訪問
    • 用於自動化配置和安裝
    • 僅在實例首次啟動時執行

置放群組

  • 集群
    • 將實例緊密打包在一起，位於同一可用區內
    • 提供最低的網絡延遲和最高的網絡吞吐量
    • 適合需要節點間高性能通信的應用
    • 例如：HPC、大數據處理
  • 分布
    • 將實例分布在不同的底層硬件上
    • 減少相關硬件故障的風險
    • 適合需要高可用性的關鍵應用
    • 每個可用區最多支持 7 個實例
  • 分區
    • 將實例分布在不同的邏輯分區上，每個分區有獨立的機架
    • 減少相關硬件故障的影響
    • 適合分布式大數據應用
    • 例如：HDFS、HBase、Cassandra

考試要點：了解不同置放群組的特點和使用場景，特別是在需要低延遲或高可用性的架構中的應用。

Auto Scaling

Auto Scaling 組

  • 定義最小、最大和期望的實例數量
    • 最小容量：保證的實例數量
    • 最大容量：可以擴展到的最大實例數量
    • 期望容量：當前應該運行的實例數量
  • 跨多個可用區以提高可用性
    • 在可用區之間平衡分配實例
    • 提高容錯能力
  • 支持多種擴展策略
    • 可以結合使用多種策略
    • 基於指標、計劃或預測進行擴展

擴展策略

  • 目標跟踪擴展
    • 基於特定指標的目標值
    • 自動調整容量以保持目標值
    • 例如：保持 CPU 利用率在 70%
    • 最簡單的設置方式
  • 步進擴展
    • 基於警報閾值的增量調整
    • 可以設置不同的調整幅度
    • 例如：CPU > 70% 時添加 2 個實例，CPU > 85% 時添加 3 個實例
  • 簡單擴展
    • 基於單一指標的調整
    • 在擴展活動後有冷卻期
    • 不如步進擴展靈活
  • 預測擴展
    • 基於歷史模式預測未來需求
    • 提前擴展以應對可預測的負載變化
    • 適合有規律負載模式的應用
  • 計劃擴展
    • 按預定時間表調整容量
    • 適合可預測的負載變化
    • 例如：工作日早上增加容量，晚上減少容量

最佳實踐：結合使用多種擴展策略可以提供更好的性能和成本效益。例如，使用預測擴展處理可預測的負載模式，使用目標跟踪擴展處理意外的負載變化。

啟動模板和配置

  • 定義 Auto Scaling 組啟動的實例配置
    • 包括 AMI、實例類型、安全組、用戶數據等
    • 可以在多個 Auto Scaling 組中重用
  • 啟動模板支持版本控制和更多功能
    • 可以創建多個版本
    • 支持實例元數據選項
    • 支持競價型實例請求
    • 推薦使用啟動模板而不是啟動配置

生命週期鉤子

  • 在實例啟動或終止過程中執行自定義操作
    • 可以在實例啟動前後執行操作
    • 可以在實例終止前執行操作
    • 可以暫停實例生命週期
  • 用途
    • 數據備份：在終止前備份數據
    • 日誌收集：在終止前收集日誌
    • 自定義配置：在啟動後進行配置
    • 與外部系統集成：通知外部系統

考試要點：了解生命週期鉤子的使用場景，特別是在需要在實例啟動或終止時執行自定義操作的情況下。

彈性負載均衡 (ELB)

負載均衡器類型

  • 應用負載均衡器 (ALB)
    • 第 7 層負載均衡器，適用於 HTTP/HTTPS 流量
    • 支持基於路徑和主機的路由
    • 支持微服務和容器
    • 支持 WebSocket 和 HTTP/2
    • 與 AWS WAF 集成
    • 支持固定會話
  • 網絡負載均衡器 (NLB)
    • 第 4 層負載均衡器，適用於 TCP/UDP/TLS 流量
    • 超低延遲，每秒可處理數百萬請求
    • 支持靜態 IP 和彈性 IP
    • 支持 PrivateLink
    • 適合需要極高性能的應用
  • 經典負載均衡器 (CLB)
    • 舊一代負載均衡器，支持第 4 層和第 7 層
    • 功能較少，建議使用 ALB 或 NLB
    • 僅用於現有應用
  • 網關負載均衡器 (GWLB)
    • 第 3 層負載均衡器，適用於虛擬設備
    • 部署、擴展和管理第三方虛擬設備
    • 例如：防火牆、入侵檢測系統、深度包檢測系統
    • 使用 GENEVE 協議

選擇指南：根據應用需求選擇合適的負載均衡器類型。需要基於內容路由選擇 ALB，需要極高性能或固定 IP 選擇 NLB，需要部署虛擬網絡設備選擇 GWLB。

健康檢查

  • 定期檢查實例健康狀態
    • 支持 HTTP、HTTPS、TCP 健康檢查
    • 可以設置檢查間隔和超時
    • 可以設置健康閾值和不健康閾值
  • 自動從負載均衡中移除不健康的實例
    • 防止將流量發送到不健康的實例
    • 實例恢復健康後自動重新加入
  • 支持自定義健康檢查路徑和閾值
    • 可以檢查應用特定的端點
    • 可以設置自定義響應代碼

粘性會話

  • 將用戶會話綁定到特定實例
    • 確保用戶請求發送到同一實例
    • 適合有狀態應用
  • 基於應用程序或負載均衡器生成的 cookie
    • 應用程序生成的 cookie：應用程序控制粘性持續時間
    • 負載均衡器生成的 cookie：ELB 控制粘性持續時間
  • ALB 和 CLB 支持粘性會話，NLB 通過源 IP 支持粘性

跨區域負載均衡

  • 在多個區域之間分配流量
    • ALB 和 CLB 支持跨可用區負載均衡
    • 可以啟用或禁用
    • 默認啟用
  • 提高可用性和災難恢復能力
    • 即使一個可用區出現故障，應用仍然可用
    • 流量自動重新分配到健康的可用區
  • 跨區域負載均衡與 Route 53 結合使用可以實現跨區域高可用性

考試要點：了解不同負載均衡器類型的特點和使用場景，以及如何配置健康檢查和粘性會話以提高應用的可用性和用戶體驗。

容器服務

Amazon ECS (彈性容器服務)

  • 完全托管的容器編排服務
    • 支持 Docker 容器
    • 與 AWS 服務深度集成
    • 可以在 EC2 或 Fargate 上運行
  • 關鍵概念
    • 集群：EC2 實例或 Fargate 容量的邏輯分組
    • 任務定義：容器的藍圖，定義容器映像、CPU、內存等
    • 任務：任務定義的實例，代表正在運行的容器
    • 服務：維護指定數量的任務
  • 啟動類型
    • EC2：在自管理的 EC2 實例上運行容器
    • Fargate：無需管理底層基礎設施
  • 與其他 AWS 服務集成
    • ALB/NLB：負載均衡
    • IAM：安全和訪問控制
    • CloudWatch：監控和日誌
    • ECR：容器映像存儲

Amazon EKS (彈性 Kubernetes 服務)

  • 托管的 Kubernetes 服務
    • 標準 Kubernetes API
    • 與現有 Kubernetes 工具兼容
    • 可以在 EC2、Fargate 或本地運行
  • 關鍵特性
    • 托管控制平面：自動升級和修補
    • 多可用區高可用性
    • 與 AWS 服務集成
    • 支持 Kubernetes 插件
  • 部署選項
    • EC2：自定義和控制節點
    • Fargate：無需管理節點
    • EKS Anywhere：在本地運行 EKS
    • EKS Distro：自行托管 Kubernetes
  • 網絡和安全
    • VPC CNI 插件
    • Pod 安全策略
    • IAM 角色用於服務賬戶
    • 私有集群支持

選擇指南：如果已經使用 Kubernetes 或需要更多控制和可移植性，選擇 EKS；如果希望更簡單的容器編排和更深入的 AWS 集成，選擇 ECS。

AWS Fargate

  • 無服務器容器計算引擎
    • 無需管理底層基礎設施
    • 按需付費，按容器級別計費
    • 與 ECS 和 EKS 兼容
  • 優勢
    • 減少運維負擔
    • 按需擴展
    • 提高安全性（每個任務隔離）
    • 精確的資源控制
  • 限制
    • 啟動時間較長
    • 不支持 GPU 工作負載
    • 不支持特定的 Kubernetes 功能
    • 成本可能高於 EC2

Amazon ECR (彈性容器註冊表)

  • 完全托管的容器映像註冊表
    • 存儲、管理和部署容器映像
    • 與 ECS 和 EKS 集成
    • 支持私有和公共存儲庫
  • 特性
    • 高可用性和可擴展性
    • 與 IAM 集成進行訪問控制
    • 映像掃描和生命週期策略
    • 跨區域和跨賬戶複製
  • 安全性
    • 傳輸中和靜態加密
    • 映像簽名和驗證
    • 漏洞掃描
    • 細粒度的訪問控制

考試要點：了解容器服務的特點和使用場景，特別是 ECS 和 EKS 的區別，以及何時選擇 Fargate 而不是 EC2 啟動類型。

無服務器計算

AWS Lambda

  • 無服務器計算服務
    • 運行代碼而無需管理服務器
    • 按使用量付費（執行時間和內存）
    • 自動擴展
    • 支持多種編程語言
  • 關鍵概念
    • 函數：執行代碼的基本單位
    • 事件源：觸發函數的服務或資源
    • 觸發器：將事件源映射到函數
    • 層：可重用的代碼和庫
  • 限制和配置
    • 執行時間：最長 15 分鐘
    • 內存：128MB 到 10GB
    • 臨時存儲：512MB 到 10GB
    • 並發執行：1000（可提高）
  • 集成和用例
    • API Gateway：創建 REST API
    • S3：處理對象事件
    • DynamoDB：處理數據庫事件
    • EventBridge：處理計劃事件
    • SQS/SNS：處理消息和通知

最佳實踐：將 Lambda 函數設計為小型、專注的單一功能，遵循單一職責原則。使用環境變量存儲配置，使用層共享代碼，使用 X-Ray 進行調試和性能分析。

AWS Step Functions

  • 無服務器工作流服務
    • 協調多個 AWS 服務
    • 使用視覺化工作流（狀態機）
    • 支持順序、並行、條件和錯誤處理
  • 工作流類型
    • 標準工作流：最長運行時間 1 年，精確一次執行
    • 快速工作流：最長運行時間 5 分鐘，至少一次執行
  • 狀態類型
    • 任務：執行工作（Lambda、ECS 任務等）
    • 選擇：條件分支
    • 並行：並行執行分支
    • 映射：迭代數組
    • 等待：暫停執行
    • 成功/失敗：結束執行
  • 用例
    • 數據處理工作流
    • IT 自動化
    • 微服務協調
    • 機器學習管道

考試要點：了解 Lambda 和 Step Functions 的集成方式，以及如何使用 Step Functions 構建複雜的無服務器工作流。
