存儲服務

存儲服務是 AWS 的基礎服務之一，SAP 考試中涉及 S3、EBS、EFS、Storage Gateway 等關鍵知識點，以及如何設計高效、安全、經濟的存儲架構。

Amazon S3 (簡單存儲服務)

基本概念

  • 對象存儲服務
    • 高度可擴展、高可用性、高耐用性
      • 可擴展性：無需預先規劃容量，可從幾KB到幾PB無縫擴展
      • 高可用性：設計為99.99%的可用性，數據分佈在至少3個可用區
      • 高耐用性：設計為99.999999999%（11個9）的耐用性，意味著存儲10,000,000個對象，平均每10,000年可能只丟失一個對象
    • 存儲無限量的數據：單個S3賬戶可以存儲無限數量的對象，沒有總容量限制
    • 適合非結構化數據：如圖片、視頻、日誌文件、備份、靜態網站內容等不需要關係型數據庫的數據
    • 支持任何類型的文件：文檔、圖片、視頻、執行文件等任何格式的數據都可以存儲
  • 存儲桶和對象
    • 存儲桶（Bucket）：
      • 存儲對象的容器，類似於文件系統中的頂級文件夾
      • 全球唯一名稱：每個存儲桶名稱在所有AWS區域和所有AWS賬戶中必須唯一
      • 區域特定：創建在特定AWS區域，數據不會自動跨區域複製
      • 無限對象：單個存儲桶可以存儲無限數量的對象
      • 命名規則：3-63個字符，只能包含小寫字母、數字、點(.)和連字符(-)
    • 對象（Object）：
      • 存儲在存儲桶中的基本實體，由數據和元數據組成
      • 元數據包括：內容類型、最後修改日期、自定義用戶元數據等
      • 不支持修改：對象一旦創建就不能修改，只能替換或刪除
    • 對象大小：
      • 單個對象最大 5TB
      • 單次上傳最大 5GB，超過需使用分段上傳
      • 最小對象大小：0字節
    • 對象鍵（Key）：
      • 唯一標識存儲桶中對象的名稱，如 "photos/2023/summer/beach.jpg"
      • 最大長度：1,024字節
      • 可以包含"/"字符模擬文件夾結構，但S3實際上是平面結構
      • 鍵名設計會影響性能：隨機前綴可提高性能
  • 數據一致性模型
    • 所有區域的所有操作都是強一致性：
      • 自2020年12月起，S3提供所有操作的強一致性（之前是最終一致性）
      • 強一致性意味著一旦寫入操作完成，所有後續的讀取請求將立即返回最新的數據
      • 無需額外配置或成本即可獲得強一致性
    • 新對象的 PUT 請求後立即可讀：
      • 上傳新對象後，可以立即讀取，無需等待數據傳播
      • 例如：上傳image.jpg後，立即執行GET請求將返回剛上傳的文件
    • 覆蓋和刪除現有對象的請求後立即可見：
      • 更新或刪除對象後，所有後續請求將立即看到更改
      • 例如：將文件從v1更新到v2後，所有後續讀取將獲得v2版本
      • 刪除對象後，對象將立即不可見
    • 原子操作：
      • PUT和DELETE操作是原子的 - 請求要麼完全成功，要麼完全失敗
      • 不會出現部分更新的情況

存儲類別

  • S3 標準（S3 Standard）
    • 默認存儲類別，高可用性和耐用性
      • 99.99% 可用性
      • 99.999999999%（11個9）耐用性
      • 設計為承受兩個設施同時損失
    • 適合頻繁訪問的數據：
      • 適用場景：網站內容、移動和遊戲應用程序、大數據分析
      • 無最低存儲時間要求，無最小對象大小限制
      • 無檢索費用，只收取存儲和請求費用
    • 低延遲、高吞吐量：
      • 毫秒級訪問延遲
      • 可支持每秒數千次請求
    • 支持多可用區域冗餘：
      • 數據自動存儲在至少三個可用區
      • 即使整個可用區故障，數據仍然可用
    • 實際應用示例：
      • 電子商務網站的產品圖片和視頻
      • 內容分發的媒體文件
      • 移動應用的用戶生成內容
  • S3 智能分層（S3 Intelligent-Tiering）
    • 自動優化成本：
      • 自動監控每個對象的訪問模式
      • 可節省高達40%的存儲成本
      • 適合訪問模式未知或不斷變化的數據
    • 根據訪問模式在不同訪問層之間移動對象：
      • 頻繁訪問層：與S3標準相同的性能和費率
      • 不頻繁訪問層：30天未訪問的對象自動移至此層，存儲成本降低
      • 歸檔即時訪問層：90天未訪問的對象（需啟用）
      • 歸檔訪問層：180天未訪問的對象（需啟用）
      • 深度歸檔訪問層：365天未訪問的對象（需啟用）
    • 無性能影響：
      • 所有層都提供與S3標準相同的低延遲和高吞吐量性能
      • 對象移動在後台自動進行，不影響應用程序
    • 無檢索費用：
      • 訪問任何層的對象都不收取檢索費用
      • 只有小額監控和自動分層費用
    • 實際應用示例：
      • 長期數據存儲，如備份和災難恢復
      • 數據湖和大數據分析存儲
      • 媒體資產庫，訪問模式不可預測
  • S3 標準 - 不頻繁訪問 (S3 Standard-IA)
    • 適合不經常訪問但需要快速訪問的數據：
      • 設計用於每月訪問一次或更少的數據
      • 需要時仍能提供毫秒級訪問
      • 與S3標準相同的耐用性和吞吐量
    • 比標準存儲成本低，但有檢索費用：
      • 存儲成本比S3標準低約40%
      • 每GB檢索費用（當訪問數據時收取）
      • 每GB數據傳輸費用（與標準相同）
    • 適合備份、長期存儲：
      • 災難恢復文件
      • 舊數據備份
      • 較舊的媒體文件存檔
    • 最低存儲持續時間：30 天
      • 刪除、覆蓋或轉換到其他存儲類別前，對象必須在此類別中至少存儲30天
      • 提前刪除仍需支付完整30天的費用
    • 最小對象大小：128KB
      • 小於128KB的對象將按128KB計費
      • 不適合存儲大量小文件
    • 實際應用示例：
      • 合規性存檔數據
      • 舊版應用程序數據
      • 較少訪問的報告和分析數據
  • S3 單區域 - 不頻繁訪問 (S3 One Zone-IA)
    • 數據存儲在單個可用區：
      • 不提供多可用區彈性
      • 如果可用區不可用或被摧毀，數據可能丟失
      • 仍提供99.5%的可用性
    • 比 S3 Standard-IA 成本低 20%：
      • 比S3標準成本低約60%
      • 與Standard-IA相同的檢索費用
      • 成本降低是因為減少了跨可用區的數據複製
    • 適合可重新創建的數據：
      • 數據副本或可從其他來源重新生成的數據
      • 數據丟失不會造成嚴重業務影響的場景
      • 非關鍵數據的二次備份
    • 最低存儲持續時間：30 天
      • 與Standard-IA相同的最低存儲期限
      • 提前刪除仍需支付完整30天的費用
    • 最小對象大小：128KB
      • 小於128KB的對象將按128KB計費
    • 實際應用示例：
      • 可重新處理的數據轉換結果
      • 縮略圖或其他衍生媒體資產
      • 已存在主備份的次要備份
  • S3 Glacier 即時檢索（S3 Glacier Instant Retrieval）
    • 適合每季度訪問一次的歸檔數據：
      • 設計用於長期存儲但偶爾需要立即訪問的數據
      • 適合每季度訪問一次或更少的數據
      • 與S3標準相同的耐用性（11個9）
    • 毫秒級檢索時間：
      • 與S3標準相同的訪問速度
      • 無需提前請求或等待即可訪問
      • 適合對時間敏感的歸檔數據
    • 比 S3 Standard-IA 成本低：
      • 存儲成本比S3標準低約68%
      • 比Standard-IA低約10%
      • 檢索費用比Standard-IA高
    • 最低存儲持續時間：90 天
      • 提前刪除仍需支付完整90天的費用
      • 適合長期存儲計劃
    • 最小對象大小：128KB
      • 小於128KB的對象將按128KB計費
    • 實際應用示例：
      • 醫療影像歸檔
      • 媒體資產長期存檔
      • 需要偶爾立即訪問的監管合規數據
  • S3 Glacier 靈活檢索（S3 Glacier Flexible Retrieval）
    • 適合長期歸檔，偶爾需要檢索：
      • 設計用於很少訪問且可以容忍檢索延遲的數據
      • 適合每年訪問1-2次的數據
      • 與其他S3類別相同的耐用性（11個9）
    • 檢索時間：提供三種檢索選項
      • 加急檢索：1-5分鐘（費用最高）
      • 標準檢索：3-5小時（平衡成本和速度）
      • 批量檢索：5-12小時（最低成本）
    • 比即時檢索成本低：
      • 存儲成本比S3標準低約80%
      • 比Glacier即時檢索低約30%
      • 檢索費用根據檢索類型而異
    • 最低存儲持續時間：90 天
      • 提前刪除仍需支付完整90天的費用
    • 最小對象大小：40KB
      • 小於40KB的對象將按40KB計費
    • 實際應用示例：
      • 長期備份和歸檔
      • 歷史數據分析存儲
      • 合規性和法規存檔
      • 數字保存項目
  • S3 Glacier 深度歸檔（S3 Glacier Deep Archive）
    • 最低成本存儲類別：
      • AWS提供的最低成本存儲選項
      • 存儲成本比S3標準低約95%
      • 比Glacier靈活檢索低約50%
    • 適合很少訪問的長期保存：
      • 設計用於保留數據但幾乎從不需要訪問的場景
      • 適合每年訪問一次或更少的數據
      • 適合"存檔後忘記"的數據
    • 檢索時間：提供兩種檢索選項
      • 標準檢索：12小時內
      • 批量檢索：最長48小時
      • 不提供加急檢索選項
    • 最低存儲持續時間：180 天
      • 最長的最低存儲期限
      • 提前刪除仍需支付完整180天的費用
      • 適合長期合規性存檔
    • 最小對象大小：40KB
      • 小於40KB的對象將按40KB計費
    • 實際應用示例：
      • 長期法規合規性存檔
      • 醫療記錄長期保存
      • 媒體公司的原始內容歸檔
      • 科學數據長期保存
    • 替代磁帶存儲：
      • 設計為磁帶庫的雲替代方案
      • 無需管理物理媒體
      • 無需擔心磁帶降解

成本優化：根據數據訪問模式選擇合適的存儲類別可以顯著降低成本。使用生命週期策略自動在不同存儲類別之間轉換對象，或使用 S3 智能分層自動優化成本。

存儲類別選擇指南：

  • 頻繁訪問（每天多次）：S3 標準
  • 訪問模式不可預測：S3 智能分層
  • 每月訪問幾次：S3 標準-IA
  • 可重新生成的不頻繁訪問數據：S3 單區域-IA
  • 每季度訪問一次且需要立即訪問：S3 Glacier 即時檢索
  • 每年訪問幾次且可接受幾小時延遲：S3 Glacier 靈活檢索
  • 長期歸檔，幾乎不訪問：S3 Glacier 深度歸檔

成本節省技巧：

  • 使用S3分析工具識別訪問模式
  • 設置生命週期規則自動轉換存儲類別
  • 使用S3智能分層處理不可預測的訪問模式
  • 考慮對象大小 - 小對象使用較低層級可能不划算
  • 監控檢索費用，特別是對於歸檔類別

數據管理功能

  • 生命週期管理
    • 自動轉換存儲類別：
      • 根據對象年齡自動將對象從一個存儲類別轉移到另一個類別
      • 例如：標準 → 標準-IA(30天) → Glacier(90天) → 深度歸檔(180天)
      • 減少手動管理存儲的需求
    • 自動刪除過期對象：
      • 設置對象的過期時間，到期後自動刪除
      • 適用於臨時數據、日誌文件或有保留期限的數據
      • 可以與版本控制結合使用，只刪除特定版本
    • 基於前綴、標籤或對象大小配置規則：
      • 前綴：應用於特定"文件夾"，如 "logs/"、"backups/"
      • 標籤：基於對象標籤應用規則，如 "project=alpha"
      • 對象大小：可以為小對象和大對象設置不同的規則
    • 可以應用於整個存儲桶或對象子集：
      • 靈活配置，可以針對不同類型的數據設置不同的規則
      • 最多可以為每個存儲桶配置1,000條生命週期規則
      • 規則可以啟用或禁用，無需刪除
    • 實際應用示例：
      • 日誌文件：30天後移至IA，90天後移至Glacier，1年後刪除
      • 備份數據：60天後移至Glacier，7年後移至深度歸檔
      • 臨時文件：7天後自動刪除
  • 版本控制
    • 保留對象的多個版本：
      • 每次修改對象時創建新版本，而不是覆蓋
      • 所有版本都保存在同一存儲桶中
      • 每個版本都是完整的對象副本，不是差異存儲
      • 可以通過版本ID訪問特定版本
    • 防止意外刪除和覆蓋：
      • 刪除操作只添加刪除標記，不實際刪除對象
      • 覆蓋操作創建新版本，保留原始版本
      • 提供數據保護和審計能力
    • 可以恢復刪除的對象：
      • 刪除刪除標記或指定特定版本ID即可恢復
      • 支持意外刪除後的數據恢復
      • 無需依賴備份即可恢復數據
    • 可以與生命週期規則結合使用：
      • 自動過期或轉換特定版本
      • 可以設置規則只保留最近N個版本
      • 可以移動舊版本到低成本存儲類別
      • 控制版本存儲成本
    • MFA刪除：
      • 需要多因素認證才能永久刪除版本
      • 額外的安全層，防止意外或惡意刪除
  • 複製和複製
    • 跨區域複製 (CRR)：跨區域複製對象
      • 自動將對象從源存儲桶複製到不同區域的目標存儲桶
      • 用例：合規性要求、降低跨區域數據訪問延遲、災難恢復
      • 只複製啟用複製後創建的新對象和更新
      • 可以使用S3批處理複製現有對象
    • 同區域複製 (SRR)：在同一區域內複製對象
      • 在同一區域內的不同存儲桶之間自動複製對象
      • 用例：日誌聚合、在不同AWS賬戶間複製數據、創建生產數據的測試副本
      • 比CRR成本低，無數據傳輸費用
    • 可以更改存儲類別和所有權：
      • 複製時可以指定不同的存儲類別
      • 可以更改對象所有權到目標存儲桶擁有者
      • 可以保留或替換元數據
    • 可以與版本控制結合使用：
      • 複製所有版本或只複製最新版本
      • 保持源和目標的版本歷史同步
      • 刪除標記也會被複製
    • 複製狀態監控：
      • 通過S3控制台或CloudWatch指標監控複製狀態
      • 設置SNS通知接收複製失敗警報
      • 複製時間取決於對象大小和區域間距離
  • 對象鎖定
    • 防止對象被刪除或覆蓋：
      • WORM（一次寫入多次讀取）模型
      • 為對象設置保留期，在此期間無法修改或刪除
      • 需要在啟用版本控制的存儲桶上配置
      • 一旦為存儲桶啟用，無法禁用對象鎖定
    • 合規模式：不可覆蓋或刪除
      • 最嚴格的保護級別
      • 任何用戶（包括根用戶）都無法覆蓋或刪除
      • 保留期無法縮短，只能延長
      • 適用於需要嚴格合規性的監管數據
    • 治理模式：特殊權限可以覆蓋保護
      • 較靈活的保護級別
      • 具有特殊權限的用戶可以覆蓋保護
      • 需要s3:BypassGovernanceRetention權限
      • 適用於需要保護但偶爾需要修改的數據
    • 法定保留：防止刪除，直到明確移除
      • 獨立於保留期的額外保護層
      • 無到期日期，直到明確移除才解除保護
      • 可以與合規或治理模式結合使用
      • 適用於法律訴訟或調查期間的數據保護
    • 實際應用場景：
      • 金融記錄：SEC規則17a-4要求的不可更改存儲
      • 醫療數據：HIPAA合規性存儲
      • 法律證據：電子證據保全
      • 公共部門：政府記錄保留

安全性

  • 訪問控制
    • IAM 策略：基於身份的訪問控制
      • 附加到IAM用戶、組或角色
      • 控制身份可以對哪些S3資源執行哪些操作
      • 可以限制特定IP範圍、時間段或請求條件
      • 適用於AWS賬戶內部用戶的訪問控制
      • 示例：允許開發團隊讀取但不寫入生產存儲桶
    • 存儲桶策略：基於資源的訪問控制
      • 直接附加到S3存儲桶的JSON策略
      • 控制誰可以訪問存儲桶及其內容
      • 可以授予或拒絕跨賬戶訪問權限
      • 可以要求特定條件（如HTTPS、MFA）
      • 示例：允許公共讀取特定前綴的對象，但要求其他所有訪問使用HTTPS
    • 訪問控制列表 (ACL)：基於對象級別的訪問控制
      • 最早的S3訪問控制機制，功能有限
      • 可以單獨控制對象級別的訪問
      • AWS建議盡可能使用存儲桶策略和IAM策略
      • 新存儲桶默認啟用"阻止所有公共訪問"，禁用ACL
    • 預簽名 URL：臨時訪問
      • 生成有時間限制的URL，允許無需AWS憑證的訪問
      • URL包含臨時憑證，繼承生成URL的用戶權限
      • 有效期可設置為5分鐘到7天
      • 用例：允許用戶上傳特定文件、提供臨時下載鏈接
      • 示例：允許用戶上傳個人資料照片到特定位置
  • 加密
    • 服務器端加密 (SSE)
      • SSE-S3：使用 Amazon S3 管理的密鑰
        • 最簡單的加密選項，AWS完全管理密鑰
        • 使用AES-256強加密算法
        • 密鑰自動輪換，用戶無需管理
        • 適合需要加密但不需要控制密鑰的場景
        • 無額外費用
      • SSE-KMS：使用 AWS KMS 管理的密鑰
        • 使用AWS Key Management Service管理密鑰
        • 提供額外的控制層和審計能力
        • 可以使用AWS管理的密鑰或客戶管理的密鑰(CMK)
        • 支持密鑰輪換和訪問控制
        • 提供使用記錄，支持合規性審計
        • 有額外的KMS API調用費用
      • SSE-C：使用客戶提供的密鑰
        • 客戶提供和管理加密密鑰
        • S3使用提供的密鑰加密數據，但不存儲密鑰
        • 每次請求都必須提供相同的密鑰
        • 只能通過HTTPS訪問
        • 適合需要完全控制加密密鑰的場景
        • 客戶負責密鑰的安全和備份
    • 客戶端加密：在上傳前加密數據
      • 數據在發送到S3之前在客戶端加密
      • 可以使用AWS加密SDK或其他加密庫
      • S3完全不知道數據已加密
      • 提供端到端加密，數據在傳輸和存儲過程中始終加密
      • 客戶完全控制加密過程和密鑰管理
      • 適合最高安全要求的場景
    • 默認加密：自動加密新對象
      • 為存儲桶設置默認加密行為
      • 所有上傳到存儲桶的新對象自動加密
      • 可以選擇SSE-S3或SSE-KMS作為默認方法
      • 可以通過存儲桶策略強制要求加密
      • 自2023年1月起，所有新對象默認使用SSE-S3加密
    • 雙層加密：
      • 結合客戶端加密和服務器端加密
      • 提供多層保護
      • 適合處理高度敏感數據
  • 網絡安全
    • VPC 端點：私有訪問 S3
      • 允許VPC內的資源通過私有網絡連接訪問S3
      • 流量不經過公共互聯網，提高安全性
      • 減少數據傳輸費用和延遲
      • 支持使用端點策略限制訪問
      • 適合需要高安全性的企業應用
    • 存儲桶策略限制 IP 和 VPC 端點
      • 可以限制只允許特定IP範圍訪問存儲桶
      • 可以要求流量只能通過特定VPC端點
      • 結合條件鍵aws:SourceIp和aws:SourceVpce
      • 防止未授權網絡訪問數據
      • 示例：只允許公司辦公網絡IP訪問敏感數據
    • 阻止公共訪問設置
      • 賬戶級別和存儲桶級別的安全控制
      • 防止意外公開存儲桶或對象
      • 可以阻止新的公共ACL、存儲桶策略或現有公共訪問
      • 默認為所有新存儲桶啟用
      • AWS建議為所有存儲桶啟用，除非明確需要公共訪問
    • 訪問日誌和監控
      • 服務器訪問日誌：記錄對存儲桶的所有請求
      • CloudTrail：記錄S3 API調用
      • CloudWatch：監控S3指標和設置警報
      • Macie：自動發現和保護敏感數據

考試要點：了解不同存儲類別的特點和成本，以及如何使用生命週期策略優化成本。理解 S3 的安全機制，特別是不同加密選項和訪問控制方法。

S3安全最佳實踐：

  • 使用阻止公共訪問設置防止意外公開數據
  • 實施最小權限原則，只授予必要的訪問權限
  • 使用IAM角色而非長期憑證進行訪問
  • 為所有敏感數據啟用加密
  • 使用VPC端點保持流量在AWS網絡內
  • 啟用訪問日誌和CloudTrail進行審計
  • 定期審查存儲桶策略和權限
  • 使用S3對象鎖定保護關鍵數據
  • 考慮使用多因素認證(MFA)刪除

性能優化

  • 請求速率
    • 每個前綴每秒 3,500 個 PUT/COPY/POST/DELETE 請求
      • 前綴是對象鍵名稱中斜杠之前的部分，如"photos/"
      • 寫入操作的基準性能限制
      • 超過此限制會導致HTTP 503錯誤（服務不可用）
    • 每個前綴每秒 5,500 個 GET/HEAD 請求
      • 讀取操作的基準性能限制
      • 讀取性能高於寫入性能
      • HEAD請求只返回元數據，不返回對象內容
    • 通過使用隨機前綴或鍵名稱分區提高性能
      • 使用隨機或均勻分佈的前綴，如哈希值、日期等
      • 例如：使用"2023-03-18/user123/photo.jpg"而非"user123/2023-03-18/photo.jpg"
      • 避免使用順序前綴（如時間戳）處理高流量工作負載
      • 可以將請求分散到多個分區，實現更高的總體吞吐量
      • 示例：使用十六進制哈希前綴(0-f)可以提供16倍的性能
    • 分區策略示例：
      • 反向時間戳：將時間戳倒序作為前綴
      • 哈希分區：使用對象ID的哈希值作為前綴
      • 隨機字符：在鍵名前添加隨機字符
      • 並行前綴：將工作負載分散到多個前綴
  • 傳輸加速
    • 使用 CloudFront 的全球分布式邊緣節點
      • 利用Amazon的全球網絡基礎設施
      • 數據通過最優路徑傳輸到S3
      • 在200多個邊緣位置提供接入點
      • 使用優化的網絡協議和路由
    • 加速上傳和下載
      • 可提高50-500%的傳輸速度
      • 距離S3區域越遠，改善效果越明顯
      • 特別適合跨大洲傳輸
      • 支持所有S3功能和API
    • 適合跨地理位置傳輸大文件
      • 全球分布的用戶上傳內容
      • 定期在不同區域間傳輸GB或TB級數據
      • 無法使用AWS Direct Connect的場景
      • 需要最大化可用互聯網帶寬
    • 啟用和使用：
      • 為存儲桶啟用傳輸加速
      • 使用特殊端點：bucketname.s3-accelerate.amazonaws.com
      • 有額外費用，基於傳輸的數據量
      • 可以使用速度比較工具測試效果
  • 多部分上傳
    • 並行上傳對象的多個部分：
      • 將大文件分割成多個較小的部分（5MB-5GB）
      • 每個部分可以獨立並行上傳
      • 所有部分上傳完成後，S3將它們組合成一個完整對象
      • 最多可以使用10,000個部分，支持高達5TB的對象
    • 提高吞吐量和恢復能力：
      • 並行上傳多個部分可以充分利用可用帶寬
      • 單個部分上傳失敗只需重試該部分，而非整個文件
      • 可以暫停和恢復上傳過程
      • 網絡條件不佳時特別有用
    • 推薦用於大於 100MB 的對象：
      • 對於大於100MB的文件，多部分上傳效率更高
      • 對於大於5GB的文件，必須使用多部分上傳
      • 可以通過AWS SDK、CLI或直接使用REST API實現
      • 許多S3客戶端工具自動處理多部分上傳
    • 實施考慮：
      • 選擇適當的部分大小：通常在25MB-100MB之間平衡性能
      • 處理未完成的上傳：設置生命週期規則清理未完成的分段上傳
      • 使用校驗和驗證數據完整性
      • 考慮使用AWS Transfer Acceleration加速上傳
  • S3 Select
    • 僅檢索對象的一部分：
      • 使用SQL表達式從對象中選擇特定數據
      • 無需下載整個對象即可提取所需信息
      • 支持CSV、JSON和Parquet格式文件
      • 可以選擇特定列、行或應用簡單條件
    • 減少傳輸的數據量：
      • 只傳輸查詢結果，而非整個對象
      • 可以將傳輸的數據量減少高達80-90%
      • 降低網絡使用和數據傳輸成本
      • 特別適合處理大型數據集
    • 提高查詢性能：
      • 查詢處理在S3服務端執行，利用AWS基礎設施
      • 可以將應用程序性能提高多達400%
      • 減少客戶端CPU和內存使用
      • 降低應用程序複雜性
    • 使用場景：
      • 數據分析：從大型日誌文件中提取特定事件
      • 報告生成：從數據集中提取特定時間段的數據
      • 數據轉換：在數據處理管道中過濾和轉換數據
      • 移動應用：減少移動設備下載的數據量

Amazon EBS (彈性塊存儲)

EBS是一種高性能、持久的塊存儲服務，專為EC2實例設計。它提供低延遲、高可用性的存儲卷，可以獨立於EC2實例的生命週期存在，並可以輕鬆地從一個實例分離並附加到另一個實例。

卷類型

  • 通用型 SSD (gp2/gp3)
    • 平衡價格和性能：
      • 提供成本效益高的存儲，適合大多數工作負載
      • 單位成本低於預置IOPS卷
      • 提供可預測的性能和一致的低延遲
    • 適合啟動卷、開發和測試環境：
      • 適用於啟動卷、開發環境、低延遲交互式應用
      • 適合中小型數據庫
      • 適合虛擬桌面、CRM和ERP應用
    • gp3：獨立配置 IOPS 和吞吐量
      • 基準性能：3,000 IOPS和125 MiB/s吞吐量
      • 可以獨立配置高達16,000 IOPS和1,000 MiB/s吞吐量
      • 性能不與卷大小掛鉤
      • 比gp2低20%的價格
      • 卷大小：1 GiB - 16 TiB
    • gp2：性能與卷大小掛鉤
      • 基準性能：每GiB 3 IOPS，最低100 IOPS
      • 最大16,000 IOPS（需要5,334 GiB或更大卷）
      • 吞吐量最高250 MiB/s
      • 突發信用機制：小卷可以突發至3,000 IOPS
      • 卷大小：1 GiB - 16 TiB
  • 預置 IOPS SSD (io1/io2)
    • 高性能、低延遲：
      • 專為I/O密集型工作負載設計
      • 提供持續的IOPS性能和一致的低延遲
      • 適合需要高達64,000 IOPS或4,000 MiB/s吞吐量的應用
    • 適合 I/O 密集型工作負載：
      • 關鍵業務數據庫（MongoDB、Oracle、Microsoft SQL Server等）
      • 延遲敏感型應用
      • 大型SAP、Oracle和Microsoft應用
    • io2 Block Express：最高性能
      • 下一代架構，專為關鍵任務工作負載設計
      • 高達256,000 IOPS
      • 高達4,000 MiB/s吞吐量
      • 亞毫秒級延遲
      • 99.999%耐用性（比io1/io2標準版高）
      • 卷大小：4 GiB - 64 TiB
    • io1/io2標準規格：
      • 可配置IOPS：每卷100-64,000 IOPS
      • io2提供比io1更高的耐用性（99.999%對99.9%）
      • io2提供更好的IOPS:GiB比率（500:1對50:1）
      • 卷大小：4 GiB - 16 TiB
    • 支持多附加（僅限 io1/io2）：
      • 允許將單個EBS卷同時附加到同一可用區中的多個EC2實例
      • 每個卷最多可附加到16個實例
      • 適用於集群文件系統（如OCFS2、GFS2）
      • 提高應用程序可用性和寫入吞吐量
  • 吞吐量優化型 HDD (st1)
    • 低成本 HDD 卷：
      • 比SSD卷成本低，適合需要高吞吐量的場景
      • 設計用於頻繁訪問、大數據量的工作負載
      • 基於磁性存儲而非固態存儲
    • 適合頻繁訪問、吞吐量密集型工作負載：
      • 大數據工作負載：Hadoop、數據倉庫、日誌處理
      • 流媒體工作負載：媒體處理、實時數據攝取
      • 需要連續讀寫的應用
    • 性能特點：
      • 基準吞吐量：每TiB 40 MiB/s
      • 突發吞吐量：每TiB 250 MiB/s
      • 最大吞吐量：500 MiB/s
      • IOPS上限：每TiB 500 IOPS，最大12,000 IOPS
      • 卷大小：125 GiB - 16 TiB
    • 不能用作啟動卷：
      • 只能作為數據卷使用，不支持操作系統啟動
      • 需要與SSD啟動卷配合使用
      • 適合作為大容量數據存儲
  • 冷 HDD (sc1)
    • 最低成本 HDD 卷：
      • EBS提供的最低成本存儲選項
      • 比st1便宜約20%，比SSD卷便宜更多
      • 適合需要大容量但訪問頻率低的數據
    • 適合不頻繁訪問的工作負載：
      • 冷數據存儲：很少訪問的歷史數據
      • 歸檔數據：需要保留但很少需要訪問的數據
      • 最低成本存儲場景：成本優先於性能的場景
    • 性能特點：
      • 基準吞吐量：每TiB 12 MiB/s
      • 突發吞吐量：每TiB 80 MiB/s
      • 最大吞吐量：250 MiB/s
      • IOPS上限：每TiB 250 IOPS，最大6,000 IOPS
      • 卷大小：125 GiB - 16 TiB
    • 不能用作啟動卷：
      • 只能作為數據卷使用，不支持操作系統啟動
      • 需要與SSD啟動卷配合使用
      • 適合作為歸檔或備份存儲

選擇指南：根據工作負載的 IOPS、吞吐量和成本需求選擇合適的 EBS 卷類型。對於數據庫等 I/O 密集型應用，選擇 io1/io2；對於大數據處理等吞吐量密集型應用，選擇 st1；對於一般用途，選擇 gp2/gp3。

EBS卷類型選擇決策樹：

  • 需要啟動卷或需要SSD性能？
    • 是 → 需要超過16,000 IOPS或需要多附加功能？
      • 是 → 選擇io1/io2（需要最高性能選擇io2 Block Express）
      • 否 → 選擇gp2/gp3（需要獨立配置IOPS和吞吐量選擇gp3）
    • 否 → 需要頻繁訪問數據？
      • 是 → 選擇st1（吞吐量優化型HDD）
      • 否 → 選擇sc1（冷HDD，最低成本）

性能和成本比較：

  • 最高性能（降序）：io2 Block Express > io2 > io1 > gp3 > gp2 > st1 > sc1
  • 最低成本（升序）：sc1 < st1 < gp3 < gp2 < io1 < io2 < io2 Block Express

快照和備份

  • EBS 快照
    • 卷的時間點備份：
      • 創建EBS卷的完整備份
      • 捕獲特定時間點的卷狀態
      • 可以在任何時間點創建，無需停止實例（但建議停止以確保數據一致性）
      • 可以從快照創建新卷，實現數據恢復或複製
    • 增量備份 - 只存儲變更的數據：
      • 首次快照包含卷上的所有數據
      • 後續快照只存儲自上次快照以來變更的數據塊
      • 節省存儲空間和快照創建時間
      • 刪除快照時，只刪除該快照獨有的數據
    • 存儲在 S3 中（不可直接訪問）：
      • 快照存儲在AWS管理的S3存儲中
      • 用戶無法直接在S3控制台查看或訪問這些快照
      • 通過EBS快照API或控制台管理
      • 按實際存儲的數據量收費
    • 可以跨區域複製：
      • 可以將快照複製到其他AWS區域
      • 支持災難恢復和地理冗餘
      • 可以在不同區域創建相同數據的卷
      • 跨區域數據傳輸會產生費用
    • 快照使用場景：
      • 定期備份：保護數據免受意外刪除或損壞
      • 數據遷移：將數據從一個區域移動到另一個區域
      • 卷類型轉換：從一種EBS卷類型轉換為另一種
      • 卷大小調整：創建更大或更小的卷
  • 快照生命週期管理
    • 自動創建和刪除快照：
      • 使用Amazon Data Lifecycle Manager (DLM)自動化快照管理
      • 無需手動創建和刪除快照
      • 確保定期備份關鍵數據
      • 自動刪除過期快照，控制存儲成本
    • 基於時間表或事件：
      • 時間表：設置固定間隔（如每12小時、每天或每週）
      • 可以指定快照創建的時間窗口
      • 可以基於標籤選擇要備份的卷
      • 支持多個不同的備份計劃
    • 可以設置保留策略：
      • 基於計數：保留最近N個快照
      • 基於時間：保留快照X天、週或月
      • 不同策略可應用於不同重要性的數據
      • 平衡備份覆蓋範圍和存儲成本
    • 生命週期策略示例：
      • 生產數據庫：每6小時一次快照，保留30天
      • 開發環境：每天一次快照，保留7天
      • 關鍵業務數據：每小時一次快照，保留最近24個，然後每天一個，保留30天
  • 快照歸檔
    • 將快照移至歸檔層：
      • 將不常用的快照從標準層移至低成本歸檔層
      • 適合需要長期保留但很少需要恢復的備份
      • 通過EBS快照控制台、API或AWS CLI管理
      • 最短歸檔期限：90天
    • 降低長期存儲成本：
      • 歸檔層存儲成本比標準層低約75%
      • 適合合規性和長期數據保留要求
      • 平衡存儲成本和恢復需求
      • 適合很少需要訪問的歷史數據
    • 恢復時間：24-72 小時
      • 從歸檔層恢復快照需要24-72小時
      • 恢復過程中，快照保持在歸檔層
      • 恢復完成後，快照可用於創建新卷
      • 恢復過程會產生一次性恢復費用
    • 使用場景：
      • 長期合規性存檔：需要保留7年或更長時間的數據
      • 項目完成後的數據保留
      • 季度或年度備份的長期存儲
      • 很少需要恢復的災難恢復備份
  • 快照鎖定
    • 防止意外或惡意刪除：
      • 保護重要快照免受意外或惡意刪除
      • 鎖定後的快照在保留期內無法刪除
      • 提供額外的數據保護層
      • 適合關鍵業務數據和合規性要求
    • 設置保留期：
      • 指定快照必須保留的時間段
      • 範圍從1天到100年
      • 保留期一旦設置，在合規模式下無法縮短
      • 可以延長保留期
    • 合規模式或治理模式：
      • 合規模式：最嚴格的保護，任何用戶（包括根用戶）都無法刪除或縮短保留期
      • 治理模式：提供保護，但具有特殊權限的用戶可以覆蓋保護
      • 類似於S3對象鎖定的模式
      • 根據合規性要求和靈活性需求選擇適當模式
    • 使用場景：
      • 金融和醫療行業的合規性要求
      • 法律保留和電子證據保全
      • 防止管理員錯誤刪除關鍵備份
      • 保護免受勒索軟件和惡意攻擊

高級功能

  • 多附加
    • 將單個 EBS 卷附加到多個 EC2 實例：
      • 允許最多16個EC2實例同時訪問同一個EBS卷
      • 所有實例可以同時讀取和寫入卷
      • 提供共享存儲解決方案
      • 適用於需要高可用性的集群應用
    • 僅支持 io1/io2 卷：
      • 僅適用於預置IOPS SSD卷類型
      • 不支持gp2/gp3、st1或sc1卷類型
      • io2 Block Express也支持多附加
      • 需要選擇適當的IOPS以支持多個實例的I/O需求
    • 所有實例必須在同一可用區：
      • 多附加功能僅限於單個可用區內
      • 不支持跨可用區附加
      • 適合單可用區內的高可用性架構
      • 需要考慮可用區故障的風險
    • 需要集群文件系統：
      • 必須使用支持並發訪問的集群文件系統
      • 例如：GFS2、OCFS2、VMFS等
      • 標準文件系統（如ext4、XFS）不支持多實例並發訪問
      • 集群文件系統負責協調多個實例的讀寫操作
    • 使用場景：
      • 高可用性數據庫集群
      • 共享文件系統
      • 應用程序集群
      • 需要共享存儲的容錯架構
  • 快速快照恢復 (FSR)
    • 消除從快照恢復的卷的初始性能損失：
      • 傳統上，從快照創建的新卷需要後台加載數據
      • 在數據完全加載前，訪問未加載的數據塊會導致延遲
      • FSR預熱快照數據，使新創建的卷立即達到完全性能
      • 消除了"初始性能懲罰"問題
    • 適合需要快速恢復的關鍵工作負載：
      • 生產數據庫：需要快速恢復的關鍵數據庫
      • 業務關鍵應用：不能容忍恢復延遲的應用
      • 災難恢復場景：需要快速啟動恢復環境
      • 測試/開發環境：需要快速創建生產數據副本
    • 按快照和可用區收費：
      • 每個啟用FSR的快照在每個可用區單獨收費
      • 費用基於快照大小和啟用FSR的時間
      • 可以選擇性地在特定可用區啟用
      • 建議僅為關鍵快照啟用此功能
    • 實施考慮：
      • 每個賬戶每個區域最多可以為50個快照啟用FSR
      • 可以設置FSR的保留時間，以控制成本
      • 可以通過AWS控制台、CLI或API啟用
      • 與其他EBS功能（如加密）兼容
  • 彈性卷
    • 動態修改卷類型、大小和性能：
      • 允許在不中斷服務的情況下修改EBS卷配置
      • 可以更改卷類型：如從gp2升級到gp3或io1/io2
      • 可以增加卷大小：擴展存儲容量
      • 可以調整性能參數：如IOPS和吞吐量
    • 無需停止實例或分離卷：
      • 修改過程中應用程序可以繼續運行
      • 避免了傳統方法需要的停機時間
      • 適合24/7運行的生產環境
      • 減少了維護窗口的需求
    • 只能增加卷大小，不能減小：
      • 卷大小只能增加，不能減少
      • 要減小卷大小，需要創建新的較小卷並遷移數據
      • 大小增加後，需要在操作系統中擴展文件系統
      • Linux：使用resize2fs或xfs_growfs等命令
      • Windows：使用磁盤管理擴展卷
    • 修改限制和考慮：
      • 每6小時只能修改一次卷配置
      • 增加大小後，需要支付更大容量的費用
      • 某些修改可能需要初始化過程，可能暫時影響性能
      • 建議在非高峰時段進行修改
    • 使用場景：
      • 應對存儲需求增長
      • 優化存儲性能
      • 降低存儲成本（如從io1遷移到gp3）
      • 根據工作負載變化調整存儲配置
  • 加密
    • 使用 AWS KMS 密鑰加密：
      • EBS卷使用AWS Key Management Service (KMS)進行加密
      • 支持AWS管理的密鑰(aws/ebs)或客戶管理的密鑰(CMK)
      • 使用AES-256加密算法
      • 加密在EC2主機和EBS存儲之間進行，確保數據在傳輸中加密
    • 加密卷的快照自動加密：
      • 從加密卷創建的快照自動加密
      • 使用與源卷相同的KMS密鑰
      • 從加密快照恢復的卷也自動加密
      • 可以在複製快照時更改KMS密鑰
    • 可以設置默認加密：
      • 可以在區域級別啟用EBS加密
      • 所有新創建的卷自動加密，無需額外配置
      • 可以使用AWS Organizations強制執行組織內的加密策略
      • 簡化了合規性和安全治理
    • 最小性能影響：
      • 加密和解密在EC2實例和EBS服務之間透明進行
      • 對I/O性能的影響極小
      • 現代EC2實例類型包含加密指令集，進一步減少性能影響
      • 無需修改應用程序代碼
    • 加密限制：
      • 不能直接更改現有卷的加密狀態
      • 需要創建快照，然後創建加密的新卷
      • 加密狀態在卷的生命週期內保持不變
      • 某些較舊或較小的EC2實例類型可能不支持加密卷

考試要點：了解不同 EBS 卷類型的性能特點和使用場景，以及如何使用快照進行備份和恢復。理解 EBS 加密和多附加等高級功能的應用場景。

EBS與EC2的關係：

  • EBS卷是獨立於EC2實例的資源，有自己的生命週期
  • 一個EBS卷一次只能附加到一個EC2實例（除非使用多附加功能）
  • EBS卷可以在EC2實例之間移動，但必須在同一可用區內
  • 要跨可用區或區域移動EBS數據，需要使用快照
  • 終止EC2實例時，默認情況下根卷會被刪除，但可以修改此行為

EBS與S3的比較：

  • EBS：塊存儲，適合需要頻繁更新的數據和操作系統；低延遲；按配置容量收費
  • S3：對象存儲，適合靜態文件和不頻繁更改的數據；可能有較高延遲；按實際使用量收費

Amazon EFS (彈性文件系統)

EFS是一種完全托管的彈性NFS文件系統服務，提供簡單、可擴展的文件存儲，可供多個EC2實例同時訪問。它是為需要共享訪問的工作負載設計的，如內容管理、Web服務、數據分析和開發環境。

基本概念

  • 完全托管的彈性 NFS 文件系統
    • 自動擴展，無需預置容量：
      • 根據使用情況自動擴展和縮減，無需手動管理容量
      • 可以從幾GB擴展到PB級，無需中斷服務
      • 按實際使用量計費，無需為未使用的容量付費
      • 消除了傳統文件系統的容量規劃和管理開銷
    • 支持 NFSv4.1 協議：
      • 使用標準的網絡文件系統協議
      • 與現有應用程序和工具兼容
      • 無需修改應用程序代碼即可使用
      • 支持標準的POSIX文件系統語義
    • 可以從多個 EC2 實例同時訪問：
      • 支持數千個EC2實例並發連接
      • 可以跨不同可用區的實例共享數據
      • 適合需要共享數據的分布式應用程序
      • 提供一致的文件系統視圖，無論從哪個實例訪問
    • 跨可用區高可用性：
      • 標準和單區域存儲類別提供不同級別的可用性
      • 標準存儲類別：數據存儲在多個可用區，提供99.999%可用性
      • 設計為在可用區故障時保持可用
      • 無需手動複製或同步數據
    • 與AWS服務集成：
      • 與AWS Backup集成進行備份管理
      • 與AWS DataSync集成進行數據遷移
      • 支持AWS IAM進行訪問控制
      • 與AWS CloudWatch集成進行監控
  • 性能模式
    • 通用模式：默認，平衡延遲和吞吐量
      • 為大多數文件系統工作負載提供最低延遲
      • 支持高達35,000個文件操作/秒
      • 適合大多數應用場景，包括Web服務、CMS和企業應用
      • 提供一致的亞毫秒級延遲
    • 最大 I/O 模式：高吞吐量，略高延遲
      • 為需要高吞吐量和並行性的工作負載設計
      • 可以支持無限數量的文件操作/秒
      • 延遲略高於通用模式
      • 適合大數據分析、媒體處理和高性能計算等場景
    • 性能模式選擇考慮：
      • 通用模式適合大多數應用，提供最佳延遲
      • 當應用需要超過35,000個文件操作/秒時，選擇最大I/O模式
      • 性能模式在創建文件系統時設置，之後無法更改
      • 如果不確定，建議從通用模式開始，根據需要遷移到新文件系統
  • 吞吐量模式
    • 突發吞吐量：基於存儲大小的吞吐量
      • 默認模式，吞吐量與文件系統大小成比例
      • 基準吞吐量：每GB 50 KiB/s的讀取，每GB 10 KiB/s的寫入
      • 所有文件系統至少有100 MiB/s的突發吞吐量
      • 可以累積突發積分，在需要時使用
      • 適合具有不規則訪問模式的工作負載
    • 預置吞吐量：獨立於存儲大小的固定吞吐量
      • 可以指定固定吞吐量，與存儲大小無關
      • 適合需要可預測性能的工作負載
      • 可以隨時增加或減少預置吞吐量
      • 按預置的吞吐量收費，無論實際使用多少
      • 適合小型文件系統但需要高吞吐量的場景
    • 彈性吞吐量：自動擴展吞吐量
      • 根據工作負載需求自動擴展和縮減吞吐量
      • 無需手動管理吞吐量
      • 適合具有不可預測訪問模式的工作負載
      • 按實際使用的吞吐量收費
      • 提供最大的靈活性和成本效益
    • 吞吐量模式選擇考慮：
      • 突發吞吐量：適合大多數一般用途工作負載，成本最低
      • 預置吞吐量：適合需要一致性能的關鍵應用
      • 彈性吞吐量：適合工作負載變化大且難以預測的場景
      • 可以隨時更改吞吐量模式，無需中斷服務

存儲類別

  • EFS 標準
    • 默認存儲類別：
      • 創建EFS文件系統時的默認選項
      • 提供最高級別的可用性和耐用性
      • 設計為99.999%的可用性（每年不到5.3分鐘的停機時間）
      • 無最低存儲期限或檢索費用
    • 適合頻繁訪問的文件：
      • 為需要低延遲和高吞吐量的工作負載設計
      • 適合活躍的生產工作負載
      • 適用於Web服務、內容管理系統、應用程序開發等
      • 提供最佳的性能特性
    • 跨多個可用區存儲數據：
      • 數據自動複製到一個區域內的多個可用區
      • 提供高可用性和耐用性
      • 在單個可用區故障時保持可用
      • 適合需要高可用性的關鍵業務應用
    • 成本考慮：
      • EFS標準的存儲成本最高
      • 適合需要頻繁訪問的數據
      • 可以與生命週期管理結合使用，自動將不常訪問的文件移至IA存儲類別
  • EFS 不頻繁訪問 (IA)
    • 適合不經常訪問的文件：
      • 設計用於存儲不頻繁訪問但需要在需要時快速訪問的數據
      • 適合備份、歸檔數據和不常訪問的內容
      • 與標準存儲類別具有相同的耐用性和可用性
      • 跨多個可用區存儲數據
    • 成本比標準低約 85%：
      • 存儲成本顯著低於標準存儲類別
      • 適合優化存儲成本的場景
      • 可以大幅降低不常訪問數據的存儲費用
      • 適合長期保留但很少訪問的數據
    • 有檢索費用：
      • 訪問IA存儲類別中的文件時收取檢索費用
      • 按檢索的數據量收費
      • 需要在成本節省和檢索費用之間權衡
      • 適合訪問頻率低於每月一次的文件
    • 生命週期管理：
      • 可以配置自動將一定時間未訪問的文件移至IA存儲類別
      • 常見策略：7、14、30或60天未訪問後轉移
      • 在文件被訪問時自動返回到標準存儲類別
      • 提供自動化的成本優化
  • EFS 單區域
    • 數據存儲在單個可用區：
      • 數據只存儲在一個可用區，而非跨多個可用區複製
      • 可用性低於標準存儲類別（99.99%對99.999%）
      • 如果可用區不可用，文件系統將不可訪問
      • 仍提供與標準存儲類別相同的耐用性
    • 比標準存儲成本低約 47%：
      • 通過消除跨可用區數據複製降低成本
      • 提供顯著的成本節省
      • 適合預算敏感的工作負載
      • 無最低存儲期限或檢索費用
    • 適合不需要多可用區彈性的工作負載：
      • 開發和測試環境
      • 非關鍵生產工作負載
      • 不需要高可用性的應用程序
  • EFS 單區域不頻繁訪問
    • 單可用區 + 不頻繁訪問：
      • 結合單區域存儲和不頻繁訪問的優勢
      • 數據只存儲在一個可用區
      • 針對不經常訪問的文件進行了優化
      • 提供最大的成本節省
    • 最低成本 EFS 存儲類別：
      • 比標準存儲類別節省高達92%的成本
      • 比EFS標準IA節省約47%
      • 比EFS單區域標準節省約85%
      • 適合極度成本敏感的場景
    • 適合不頻繁訪問且不需要多可用區彈性的數據：
      • 開發和測試環境的歸檔數據
      • 非關鍵備份和歷史數據
      • 可以重新生成或在其他地方有備份的數據
      • 訪問頻率低於每月一次的數據
    • 使用考慮：
      • 有檢索費用，按檢索的數據量收費
      • 可用性較低（99.9%），可能受單一可用區故障影響
      • 通常與生命週期管理結合使用
      • 適合作為成本優化策略的一部分

成本優化：使用生命週期管理自動將不經常訪問的文件轉移到 IA 存儲類別。如果不需要多可用區彈性，考慮使用單區域存儲類別降低成本。

EFS存儲類別選擇指南：

  • 需要高可用性和頻繁訪問：EFS標準
  • 需要高可用性但不頻繁訪問：EFS不頻繁訪問(IA)
  • 不需要高可用性但頻繁訪問：EFS單區域
  • 不需要高可用性且不頻繁訪問：EFS單區域不頻繁訪問

生命週期管理最佳實踐：

  • 根據訪問模式設置適當的轉換時間（7-90天）
  • 監控訪問模式和存儲分佈
  • 平衡檢索費用和存儲節省
  • 考慮工作負載特性和文件大小分佈

安全性

  • 網絡訪問控制
    • VPC 安全組：
      • 控制EC2實例和EFS掛載目標之間的網絡流量
      • 定義允許訪問EFS的源IP、協議和端口
      • 通常允許TCP端口2049（NFS）的入站流量
      • 可以限制特定VPC或子網的訪問
    • NFS 級別訪問控制：
      • 使用EFS文件系統策略控制NFS客戶端訪問
      • 可以基於IP地址限制訪問
      • 支持只讀訪問模式
      • 可以限制根用戶訪問（root_squash）
    • IAM 身份驗證：
      • 使用AWS IAM控制對EFS的訪問
      • 支持基於身份的訪問控制
      • 需要使用EFS掛載助手
      • 可以定義精細的訪問策略，如只讀或只寫
    • 掛載目標：
      • 在每個需要訪問的可用區創建掛載目標
      • 每個掛載目標有自己的安全組
      • 控制哪些子網可以訪問文件系統
      • 可以隨時添加或刪除掛載目標
  • 加密
    • 傳輸中加密：使用 TLS
      • 使用傳輸層安全協議(TLS)加密數據傳輸
      • 需要使用EFS掛載助手啟用
      • 保護數據在網絡傳輸過程中的安全
      • 防止中間人攻擊和竊聽
    • 靜態加密：使用 KMS 密鑰
      • 使用AWS Key Management Service(KMS)密鑰加密
      • 支持AWS管理的密鑰或客戶管理的密鑰(CMK)
      • 使用AES-256加密算法
      • 加密在文件系統級別進行，對用戶透明
    • 創建時啟用，無法更改：
      • 加密設置必須在創建文件系統時配置
      • 創建後無法更改加密設置
      • 要更改加密設置，需要創建新的文件系統並遷移數據
      • 可以設置默認加密，自動加密所有新文件系統
    • 加密最佳實踐：
      • 始終啟用靜態加密和傳輸中加密
      • 對於敏感數據，使用客戶管理的KMS密鑰
      • 實施密鑰輪換策略
      • 使用IAM策略限制密鑰使用
  • 訪問點
    • 為應用程序提供特定目錄視圖：
      • 創建文件系統的應用程序特定入口點
      • 每個訪問點可以暴露不同的目錄路徑
      • 可以為不同應用程序創建多個訪問點
      • 簡化共享文件系統的訪問管理
    • 強制執行用戶身份：
      • 可以為每個訪問點指定操作系統用戶和組ID
      • 覆蓋客戶端提供的身份信息
      • 確保應用程序以特定權限訪問文件系統
      • 簡化權限管理，無需在每個客戶端配置相同的用戶
    • 限制根目錄訪問：
      • 將應用程序限制在特定目錄內
      • 防止訪問文件系統的其他部分
      • 實現邏輯隔離，提高安全性
      • 適合多租戶環境和容器化應用
    • 使用場景：
      • 容器化應用：為每個容器提供特定視圖
      • 多租戶環境：隔離不同租戶的數據
      • 微服務架構：為每個服務提供所需的最小訪問
      • 簡化權限管理：集中管理文件系統訪問

備份和恢復

  • AWS Backup 集成
    • 自動備份 EFS 文件系統：
      • 使用AWS Backup服務創建EFS文件系統的完整備份
      • 增量備份 - 只備份自上次備份以來更改的文件
      • 支持定期自動備份，無需手動操作
      • 備份過程不影響文件系統性能和可用性
    • 設置備份計劃和保留策略：
      • 定義備份頻率：每小時、每天、每週或自定義
      • 設置備份窗口，選擇低流量時段
      • 配置保留期限：從幾天到幾年
      • 支持基於標籤的備份選擇
    • 跨區域備份：
      • 將備份複製到不同的AWS區域
      • 提供地理冗餘和災難恢復能力
      • 保護數據免受區域級故障影響
      • 支持合規性和業務連續性要求
    • 備份管理：
      • 集中管理所有EFS備份
      • 監控備份狀態和歷史
      • 支持手動和自動備份
      • 與其他AWS服務的備份集成
  • 時間點恢復
    • 恢復到特定時間點：
      • 從備份中選擇特定的時間點進行恢復
      • 可以恢復到任何成功備份的時間點
      • 精確恢復到意外刪除或損壞前的狀態
      • 支持災難恢復和數據保護場景
    • 創建新的文件系統：
      • 恢復操作創建全新的EFS文件系統
      • 原始文件系統保持不變
      • 可以選擇與源文件系統不同的配置
      • 可以恢復到不同區域，支持災難恢復
    • 恢復選項：
      • 完全恢復：恢復整個文件系統
      • 可以在恢復後使用訪問點限制訪問特定目錄
      • 支持不同存儲類別的恢復
      • 可以選擇不同的性能模式和吞吐量模式
    • 使用場景：
      • 災難恢復：從系統故障或數據損壞中恢復
      • 意外刪除：恢復誤刪的文件或目錄
      • 測試和開發：創建生產數據的副本用於測試
      • 數據遷移：將數據遷移到新的文件系統或區域

考試要點：了解 EFS 與 EBS 的區別，特別是在多實例訪問和跨可用區可用性方面。理解不同存儲類別和性能模式的適用場景，以及如何優化成本和性能。

EFS與EBS的主要區別：

  • 訪問模式：EFS支持多個EC2實例同時訪問，EBS一般只能附加到單個EC2實例（除非使用多附加）
  • 可用性：EFS標準自動跨多個可用區，EBS限於單個可用區
  • 協議：EFS使用NFS協議（文件級訪問），EBS提供塊級存儲
  • 擴展：EFS自動擴展，無需預置容量；EBS需要預先指定大小，可以後續調整
  • 性能：EBS提供更低的延遲和更可預測的性能；EFS提供更高的總吞吐量和並發性
  • 計費：EFS按實際使用量計費，EBS按配置容量計費

使用場景選擇：

  • 選擇EFS：需要共享文件存儲、內容管理、Web服務、開發環境、數據分析
  • 選擇EBS：數據庫、啟動卷、需要低延遲的應用、需要塊級存儲的應用

AWS Storage Gateway

AWS Storage Gateway是一系列混合雲存儲服務，使本地環境能夠無縫使用AWS雲存儲。它提供了本地應用程序通過標準協議訪問雲存儲的能力，同時保持低延遲訪問和與現有系統的兼容性。

類型和用例

  • S3 文件網關
    • 通過 NFS/SMB 協議訪問 S3：
      • 將S3對象存儲表示為本地文件共享
      • 支持NFS v3/v4.1和SMB協議
      • 與現有應用程序和工作流程兼容
      • 無需修改應用程序即可使用S3存儲
    • 本地緩存提供低延遲訪問：
      • 在本地緩存頻繁訪問的數據
      • 減少訪問雲存儲的延遲
      • 智能緩存算法優化性能
      • 緩存大小可配置，基於本地存儲容量
    • 適合文件共享、數據遷移：
      • 企業文件共享和協作
      • 將本地文件數據逐步遷移到雲端
      • 備份和歸檔解決方案
      • 雲端內容管理和分發
    • 主要特點：
      • 支持S3的所有存儲類別，包括S3 Glacier
      • 與S3版本控制、生命週期策略和加密集成
      • 支持Active Directory集成進行身份驗證
      • 提供文件和文件夾級別的訪問控制
  • FSx 文件網關
    • 提供對 Amazon FSx for Windows File Server 的低延遲訪問：
      • 連接本地環境與雲端FSx文件系統
      • 使用標準SMB協議
      • 支持Windows文件系統功能（ACL、DFS等）
      • 與Active Directory完全集成
    • 本地緩存常用數據：
      • 在本地緩存頻繁訪問的文件
      • 減少網絡延遲和帶寬使用
      • 提供近乎本地的訪問性能
      • 自動管理緩存，優先保留活躍數據
    • 適合 Windows 文件共享：
      • Windows應用程序和工作負載
      • 混合雲文件服務
      • 分支機構和遠程辦公室文件訪問
      • 需要Windows文件系統功能的應用
    • 主要特點：
      • 支持SMB 2.0-3.1.1協議
      • 與Windows安全模型完全兼容
      • 支持NTFS文件系統功能
      • 簡化分支機構IT基礎設施
  • 卷網關
    • iSCSI 塊存儲接口：
      • 提供標準iSCSI塊設備接口
      • 與現有塊存儲應用程序和備份軟件兼容
      • 支持兩種卷類型：緩存卷和存儲卷
      • 無需修改應用程序即可使用雲存儲
    • 數據存儲在 S3，作為 EBS 快照備份：
      • 數據最終存儲在S3中，提供持久性和可擴展性
      • 支持EBS快照功能，實現時間點恢復
      • 緩存卷：主數據存儲在S3，本地只保留活躍數據
      • 存儲卷：數據存儲在本地，定期備份到S3
    • 適合備份、災難恢復：
      • 將現有備份基礎設施連接到雲存儲
      • 提供低成本的災難恢復解決方案
      • 支持增量備份，減少數據傳輸
      • 可以在AWS中恢復卷，實現雲端故障轉移
    • 主要特點：
      • 支持高達32TB的卷大小
      • 提供本地緩存，優化性能
      • 支持卷克隆和快照
      • 與現有備份軟件和工作流程集成
  • 磁帶網關
    • 虛擬磁帶庫 (VTL) 接口：
      • 模擬物理磁帶庫、磁帶驅動器和介質更換器
      • 提供標準iSCSI VTL接口
      • 支持虛擬磁帶容量從100GiB到5TiB
      • 可以管理高達1萬個虛擬磁帶
    • 與現有備份軟件兼容：
      • 支持主流備份應用程序，如Veeam、Veritas、Commvault等
      • 無需更改現有備份流程和策略
      • 使用熟悉的備份軟件界面管理備份
      • 支持增量備份和完全備份
    • 數據存儲在 S3 和 Glacier：
      • 活動磁帶存儲在S3中，提供快速訪問
      • 歸檔磁帶存儲在S3 Glacier或Deep Archive中，降低長期存儲成本
      • 自動管理數據生命週期
      • 提供11個9的數據耐用性
    • 適合長期歸檔和替換物理磁帶：
      • 消除物理磁帶基礎設施的管理開銷
      • 降低長期數據保留的成本
      • 簡化合規性和監管數據保留
      • 消除磁帶運輸、存儲和處理的物流問題
    • 主要特點：
      • 支持行業標準LTO磁帶格式
      • 提供磁帶目錄和條形碼管理
      • 支持磁帶加密和寫保護
      • 提供磁帶庫監控和報告

部署選項

  • 虛擬機設備

      • 可以根據工作負載需求調整資源分配
    • 管理和維護：
      • 通過AWS Management Console集中管理
      • 自動軟件更新，確保最新功能和安全修復

      • 提供詳細的性能指標和日誌

      • AWS提供的專用物理服務器

      • 提供高性能本地緩存存儲
    • 預配置 Storage Gateway 軟件：
      • 出廠時已安裝最新版本的軟件
      • 簡化部署過程，減少配置錯誤
      • 支持自動更新，確保軟件始終最新
      • 包含AWS優化的配置設置
    • 適合無法運行虛擬機的環境：
      • 缺乏虛擬化基礎設施的環境
      • 對物理隔離有要求的安全環境
      • 需要專用硬件性能的高吞吐量場景
      • 簡化IT管理，減少維護負擔
    • 部署和管理：
      • 通過AWS控制台激活和管理
      • 支持機架安裝，標準1U機架尺寸
      • 包含冗餘電源和網絡接口

  • Amazon EC2 實例
    • 在 AWS 雲中運行：
      • 部署在EC2實例上，無需本地硬件
      • 使用預配置的AMI（Amazon機器映像）
      • 支持多種EC2實例類型，可根據性能需求選擇
      • 利用AWS雲的可靠性和可擴展性

      • 使用AWS Direct Connect提供專用網絡連接
      • 或通過VPN建立安全的加密連接
      • 確保數據傳輸的安全性和可靠性
      • 支持高帶寬、低延遲連接
    • 使用場景：
      • 雲端災難恢復解決方案
      • 將本地應用程序遷移到AWS
      • 在雲中處理和分析本地數據
      • 作為分支機構到雲的連接點
    • 優勢：
      • 無需管理物理或虛擬基礎設施
      • 可以利用EC2的自動擴展和高可用性功能
      • 與其他AWS服務緊密集成
      • 按需付費，無需前期硬件投資

混合雲存儲：Storage Gateway 是構建混合雲存儲架構的關鍵組件，允許本地應用程序無縫訪問 AWS 存儲服務，同時保持低延遲訪問和與現有系統的兼容性。

部署選擇考慮因素：

  • 虛擬機設備：適合已有虛擬化環境、希望靈活管理資源、需要與現有基礎設施集成的組織
  • 硬件設備：適合缺乏虛擬化專業知識、需要簡化部署、對性能有較高要求的環境
  • EC2實例：適合已經在使用AWS服務、希望減少本地基礎設施、需要靈活擴展的組織

網絡考慮：

  • 帶寬：確保有足夠的網絡帶寬支持預期的數據傳輸量
  • 延遲：評估應用程序對存儲延遲的敏感度
  • 可靠性：考慮實施冗餘連接以提高可用性
  • 安全性：使用加密和私有連接保護數據傳輸

      • 可以並行使用多個Snowmobile進行更大規模的遷移
      • 支持高達1Tbps的數據傳輸速率
    • 適合大規模數據中心遷移：
      • 設計用於單次遷移10PB以上的數據
      • 適合數據中心關閉、整合或大規模遷移到AWS
      • 比使用網絡傳輸更快、更經濟
      • 適合視頻庫、圖像存儲庫、科學數據集等大型數據集
    • 部署和操作：
      • AWS提供專業團隊處理運輸、設置和配置
      • 需要適當的場地條件和電力供應
      • 通常需要提前數週計劃和協調
      • 包括全面的安全措施和數據保護

用例

  • 大規模數據遷移
    • 將 PB 級數據遷移到 AWS：
      • 適合遷移大型數據集，如企業數據倉庫、媒體資產庫、科學數據集
      • 支持一次性遷移或分階段遷移策略
      • 提供端到端數據驗證，確保數據完整性
      • 支持加密，保護傳輸中的敏感數據
    • 適合網絡帶寬有限的情況：
      • 解決網絡帶寬限制和高延遲問題
      • 避免長時間占用生產網絡帶寬
      • 適合網絡連接不穩定或成本高昂的地區
      • 無需升級現有網絡基礎設施
    • 比互聯網傳輸更快、更經濟：
      • 數據傳輸速度計算：10PB數據通過1Gbps網絡需要約2.5年
      • 使用Snow設備可將時間縮短至數週
      • 避免高昂的網絡帶寬成本
      • 降低網絡傳輸失敗和重試的風險
    • 遷移流程：
      • AWS發送設備到客戶站點
      • 客戶連接設備並使用提供的工具傳輸數據
      • 設備返回AWS，數據上傳到指定的S3存儲桶
      • 提供端到端跟踪和狀態更新
  • 邊緣計算
    • 在網絡連接有限的位置運行應用程序：
      • 適合遠程站點、船舶、採礦設施、石油鑽井平台等
      • 支持間歇性連接或完全離線的環境
      • 提供本地計算和存儲資源
      • 可以在惡劣或極端環境中運行
    • 支持 EC2 兼容實例和 AWS Lambda：
      • 運行與AWS雲中相同的應用程序和工作負載
      • 支持EC2 AMI和容器化應用
      • 通過AWS Lambda@Edge運行無服務器函數
      • 使用熟悉的AWS工具和API進行管理
    • 可以在本地處理數據並僅發送結果：
      • 減少需要傳輸的數據量，節省帶寬
      • 支持實時分析和決策
      • 適合IoT數據處理、圖像分析、機器學習推理
      • 過濾和壓縮數據，只傳輸有價值的信息
    • 邊緣計算應用場景：
      • 製造業：工廠機器監控和預測性維護
      • 醫療保健：遠程醫療設施的醫學圖像處理
      • 能源：油田和風電場的傳感器數據分析
      • 運輸：車隊管理和自主車輛數據處理
  • 災難恢復
    • 將數據備份到 AWS：
      • 創建關鍵業務數據的離線備份
      • 支持定期備份策略，如每週或每月備份
      • 提供物理隔離的數據保護
      • 支持加密備份，確保數據安全
    • 在災難恢復站點預先部署設備：
      • 在災難恢復站點保持Snow設備，以備緊急情況
      • 支持快速數據恢復，無需依賴網絡連接
      • 可以在本地運行關鍵應用程序
      • 提供業務連續性解決方案
    • 災難恢復策略：
      • 使用Snowball Edge作為本地備份目標
      • 定期將數據傳輸到AWS進行長期保存
      • 在災難情況下，可以從AWS快速恢復數據
      • 支持混合雲災難恢復架構
    • 合規性和審計：
      • 支持數據保留和合規性要求
      • 提供數據傳輸的完整審計跟踪
      • 符合多種行業標準和法規
      • 支持加密和安全控制
  • 數據中心退役
    • 關閉數據中心時遷移數據：
      • 支持數據中心整合、關閉或遷移項目
      • 提供完整數據庫、文件系統和應用程序數據的遷移
      • 支持舊系統和遺留數據的遷移
      • 確保數據完整性和鏈接保持不變
    • 適合一次性大規模遷移：
      • 支持嚴格的遷移時間表和截止日期
      • 可以並行使用多個設備加速遷移
      • 適合需要在特定時間窗口完成的遷移
      • 減少遷移過程中的業務中斷
    • 遷移規劃和執行：
      • AWS提供遷移評估和規劃支持
      • 根據數據量和時間表確定所需設備數量
      • 協調設備交付和提貨時間表
      • 提供數據驗證和遷移完成報告
    • 成本效益：
      • 消除維護舊數據中心的持續成本
      • 避免租賃延期或設備更新費用
      • 減少電力、冷卻和設施管理成本
      • 支持"全部遷移到雲"的戰略

考試要點：了解何時使用 Snow 系列設備而不是通過互聯網傳輸數據，以及如何選擇合適的 Snow 設備類型。理解 Storage Gateway 的不同類型及其適用場景。

Snow系列設備選擇指南：

  • Snowcone：適合空間受限環境、需要便攜性、數據量小於14TB、需要邊緣計算能力的場景
  • Snowball Edge：適合中等到大型數據遷移(10-80TB)、需要強大本地計算能力、需要集群功能的場景
  • Snowmobile：適合超大規模數據遷移(>100TB)、數據中心遷移、大型媒體庫遷移的場景

何時使用Snow設備而非互聯網傳輸：

  • 數據量大於10TB時，物理傳輸通常更快
  • 可用網絡帶寬低於100Mbps時
  • 傳輸時間超過一週時
  • 網絡成本高或不穩定時
  • 需要離線數據處理能力時

安全考慮：

  • 所有Snow設備都提供硬件加密(256位)
  • 支持AWS KMS管理的密鑰
  • 設備包含防篡改功能和安全監控
  • 符合HIPAA、GDPR等合規標準
