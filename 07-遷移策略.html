遷移策略和方法

6R 遷移策略

  • 重新託管 (Rehost)
    • "直接遷移"或"搬遷升降"
    • 將應用程序移至雲端而不進行更改
    • 通常使用 AWS MGN 或 CloudEndure
    • 優點：快速、低風險、最小化變更
    • 缺點：無法充分利用雲的優勢
  • 重新平台化 (Replatform)
    • "提升和優化"
    • 進行一些雲優化而不改變核心架構
    • 例如：遷移到 RDS、使用自動擴展
    • 優點：平衡雲收益和工作量
    • 缺點：需要更多時間和技能
  • 重新購買 (Repurchase)
    • "丟棄和購買"
    • 從現有應用程序轉向 SaaS 產品
    • 例如：從自託管 CRM 轉向 Salesforce
    • 優點：減少維護負擔，使用現代功能
    • 缺點：可能需要數據遷移和培訓
  • 重新架構 (Refactor/Re-architect)
    • 重新設計應用程序以充分利用雲原生功能
    • 例如：轉向微服務、使用無服務器架構
    • 優點：最大化雲收益，提高敏捷性
    • 缺點：成本高、時間長、風險大
  • 保留 (Retain)
    • 保持應用程序在現有環境中
    • 適用於不適合遷移的應用程序
    • 例如：即將退役的應用程序、合規要求
    • 優點：避免不必要的工作
    • 缺點：錯過雲收益
  • 停用 (Retire)
    • 停用不再需要的應用程序
    • 適用於冗餘或過時的應用程序
    • 優點：減少維護負擔和成本
    • 缺點：需要數據歸檔和用戶通知

策略選擇：選擇適當的遷移策略取決於多種因素，包括業務優先級、時間限制、應用程序複雜性、技術債務和預算。通常，組織會採用混合方法，對不同的應用程序使用不同的策略。例如，對於關鍵業務應用程序可能選擇重新託管以快速遷移，然後在雲中逐步現代化。

AWS 遷移工具和服務

發現和規劃工具

  • AWS Application Discovery Service
    • 收集有關本地數據中心的信息
    • 代理式發現：詳細的系統配置和性能數據
    • 無代理發現：基於 VMware vCenter 的發現
    • 與 Migration Hub 集成
  • AWS Migration Hub
    • 集中跟踪遷移進度
    • 可視化應用程序組合
    • 與其他 AWS 遷移工具集成
    • 提供狀態更新和指標
  • AWS Migration Evaluator
    • 評估遷移業務案例
    • 提供成本估算
    • 識別節約機會
    • 生成報告

遷移工具

  • AWS Application Migration Service (MGN)
    • AWS 的主要遷移服務
    • 自動將物理、虛擬或雲服務器遷移到 AWS
    • 最小停機時間
    • 支持大多數操作系統
  • AWS Database Migration Service (DMS)
    • 遷移關係型數據庫、數據倉庫、NoSQL 數據庫
    • 支持同構和異構遷移
    • 最小停機時間
    • 持續數據複製
  • AWS Schema Conversion Tool (SCT)
    • 自動將數據庫架構從一種引擎轉換為另一種
    • 與 DMS 配合使用
    • 評估複雜性和兼容性
    • 轉換存儲過程和函數
  • AWS DataSync
    • 在本地存儲和 AWS 存儲服務之間傳輸數據
    • 高速數據傳輸
    • 自動加密和驗證
    • 支持 S3、EFS、FSx
  • AWS Transfer Family
    • 使用 SFTP、FTPS、FTP 傳輸文件
    • 無需修改客戶端應用程序
    • 與 S3 和 EFS 集成
    • 支持現有身份提供商

工具選擇：選擇合適的遷移工具取決於遷移策略、源環境和目標環境。例如，對於重新託管策略，AWS MGN 是首選工具；對於數據庫遷移，DMS 和 SCT 是關鍵工具；對於大量數據傳輸，可以考慮 DataSync 或 Snow Family 設備。

特殊遷移場景

  • VMware Cloud on AWS
    • 將 VMware 工作負載遷移到 AWS
    • 使用熟悉的 VMware 工具和流程
    • 無需重新平台化或重新架構
    • 與 AWS 服務集成
  • AWS Snow Family
    • 用於大規模數據傳輸的物理設備
    • Snowcone：小型便攜設備
    • Snowball：中型數據傳輸設備
    • Snowmobile：極大規模數據傳輸
  • AWS Mainframe Modernization
    • 將大型機應用程序遷移到 AWS
    • 重新平台化和重構選項
    • 自動化工具和流程
    • 托管運行時環境
  • AWS End-of-Support Migration Program (EMP)
    • 遷移運行在過時操作系統上的應用程序
    • 使用應用程序兼容性包
    • 無需重寫代碼
    • 減少安全風險

遷移後優化

遷移最佳實踐和常見挑戰

遷移最佳實踐

  • 建立雲卓越中心 (CCoE)

    • 管理成本和資源
    • 確保合規
  • 採用基礎設施即代碼 (IaC)
    • 使用 AWS CloudFormation 或 Terraform
    • 版本控制基礎設施
    • 自動化部署
    • 確保一致性
  • 實施持續集成/持續部署 (CI/CD)
    • 自動化測試和部署
    • 使用 AWS CodePipeline、CodeBuild、CodeDeploy
    • 減少人為錯誤
    • 加快發布速度
  • 建立全面的測試策略
    • 功能測試
    • 性能測試
    • 安全測試
    • 災難恢復測試
  • 制定詳細的切換計劃
    • 定義切換步驟
    • 建立回滾計劃
    • 設置成功標準
    • 分配角色和責任

常見遷移挑戰

  • 技能差距
    • 缺乏雲專業知識
    • 解決方案：培訓、合作夥伴支持、AWS 專業服務
  • 應用程序依賴關係
    • 複雜的依賴關係難以映射
    • 解決方案：使用發現工具、依賴關係映射、分階段遷移
  • 數據遷移
    • 大量數據傳輸
    • 解決方案：AWS DMS、DataSync、Snow Family
  • 網絡連接
    • 帶寬限制
    • 解決方案：AWS Direct Connect、VPN、Snow Family
  • 安全和合規
    • 滿足安全要求
    • 解決方案：AWS 安全服務、合規框架、第三方審計
  • 業務連續性
    • 最小化停機時間
    • 解決方案：藍/綠部署、金絲雀發布、數據複製
  • 組織變革管理
    • 抵制變革
    • 解決方案：溝通計劃、培訓、早期成功案例

風險緩解：成功的遷移需要識別和緩解風險。關鍵風險緩解策略包括：詳細的規劃和測試、分階段方法、明確的回滾計劃、充分的培訓和溝通，以及與經驗豐富的合作夥伴合作。

遷移案例研究

  • 大型企業遷移
    • 挑戰：複雜的應用程序組合、嚴格的合規要求
    • 方法：遷移工廠模型、混合策略、分階段方法
    • 結果：成本節約、敏捷性提高、創新加速
  • 數據中心退役
    • 挑戰：緊迫的截止日期、遺留系統
    • 方法：以重新託管為主、並行工作流、自動化工具
    • 結果：按時完成、避免數據中心續約成本
  • 數字化轉型
    • 挑戰：現代化老舊應用程序、提高客戶體驗
    • 方法：重新架構關鍵應用程序、採用微服務和無服務器
    • 結果：更快的創新、改善的客戶體驗、新收入流
  • 收購整合
    • 挑戰：整合不同環境、標準化流程
    • 方法：建立共享服務平台、統一身份管理
    • 結果：簡化運營、降低成本、提高協作
